CAMBIOS DE SEGURIDAD Y CORRECCIONES
Fecha: 28/03/2026

1. AUTENTICACION DEL FRONTEND

Problema detectado:
- El login, el registro y el reenvio del mail de verificacion usaban Supabase Auth directo desde el frontend.
- Eso dejaba afuera el rate limiting que se implemento en el backend.

Cambio aplicado:
- El frontend ahora usa el backend para:
  - /api/auth/login
  - /api/auth/register
  - /api/auth/verify-email/resend

Impacto:
- El flujo real que usa el usuario queda cubierto por los limites de intentos del backend.
- Los mensajes de error del rate limit ahora pueden mostrarse correctamente en la interfaz.


2. RATE LIMITING EFECTIVO

Estado final:
- Login bloquea por demasiados intentos.
- Registro bloquea por demasiados intentos.
- Reenvio de verificacion bloquea por demasiados intentos.

Mensajes expuestos al cliente:
- AUTH_LOGIN_RATE_LIMITED
  "Too many login attempts, please try again later"
- AUTH_REGISTER_RATE_LIMITED
  "Too many registration attempts, please try again later"
- AUTH_VERIFY_EMAIL_RESEND_RATE_LIMITED
  "Too many verification email requests, please try again later"

Impacto:
- Reduce brute force, abuso de formularios y spam de reenvios.


3. BORRADO REAL DE USUARIOS DESDE ADMIN

Problema detectado:
- El panel admin eliminaba solo la fila local en la tabla users.
- Si la cuenta seguia viva en Supabase Auth, el usuario podia reaparecer al volver a iniciar sesion.

Cambio aplicado:
- El borrado admin ahora intenta eliminar primero la cuenta de autenticacion real.
- Luego limpia registros relacionados del usuario en tablas de la app.

Impacto:
- El borrado desde admin pasa a ser mucho mas coherente con lo que espera quien administra.


4. RESTRICCION PRO PARA BANNER

Problema detectado:
- La restriccion "solo PRO puede subir banner" estaba apoyada sobre todo en frontend.

Cambio aplicado:
- La actualizacion de banner ahora pasa por backend.
- El backend rechaza el cambio si el usuario no es PRO.
- El helper frontend que intentaba escribir banner_url directo en Supabase ya no se usa para eso.

Impacto:
- La restriccion comercial deja de depender solo del cliente.


5. TOKENS DE VERIFICACION DE EMAIL

Problema detectado:
- Los tokens de verificacion se guardaban en texto plano.

Cambio aplicado:
- Los nuevos tokens se guardan hasheados con SHA-256.
- La verificacion sigue aceptando tokens viejos en texto plano para no romper enlaces ya emitidos.

Impacto:
- Si alguien leyera la tabla de tokens, no podria reutilizar directamente los nuevos tokens.


6. SANITIZACION DEL EMAIL HTML

Problema detectado:
- Valores controlados por usuario podian interpolarse en el HTML del correo.

Cambio aplicado:
- Se escapan los valores antes de insertarlos en la plantilla del mail.

Impacto:
- Reduce riesgo de inyeccion de markup dentro del correo transaccional.


7. DEPENDENCIA VULNERABLE DEL BACKEND

Problema detectado:
- npm audit marco una vulnerabilidad en path-to-regexp usada por el backend.

Cambio aplicado:
- Se fijo una version corregida mediante override.
- Se actualizo package-lock del backend.

Impacto:
- Se elimina el hallazgo reportado por npm audit para esa cadena de dependencias.

9. COSAS A TENER EN CUENTA

- .env no debe subirse al repositorio.
- El webhook de Mercado Pago debe seguir teniendo configurado su secreto en produccion.
- El fallback en memoria del rate limiting existe para resiliencia, pero el modo deseado en produccion es el respaldado por Supabase.


10. RESULTADO GENERAL

Estado final esperado despues de deploy:
- Login, registro y resend pasan por backend.
- El rate limiting protege el flujo real del usuario.
- El borrado admin deja de ser solo superficial.
- El banner PRO queda reforzado del lado servidor.
- Los tokens de verificacion quedan mejor protegidos.


11. CORRECCIONES FUNCIONALES POSTERIORES

Problemas detectados:
- El flujo de compra PRO enviaba al login, pero despues del inicio de sesion mandaba siempre a /profile.
- Varias ediciones de perfil podian quedar a mitad de camino si fallaba el segundo paso.
- El banner podia no reaparecer correctamente despues de login o recarga.
- La pagina de artista podia dejar reseñas y likes desactualizados al cambiar el estado de sesion.
- Si fallaba la creacion del pago PRO, la UI casi no mostraba feedback.
- La busqueda manual podia disparar requests duplicados.
- Los filtros de fecha del admin podian correrse por zona horaria.

Cambio aplicado:
- El login ahora respeta redirectTo y vuelve a /pro cuando corresponde.
- Las actualizaciones de username, telefono y avatar pasan a consolidarse desde backend.
- Si un upload de avatar o banner falla en el paso final, se intenta limpiar el archivo recien subido para no dejar basura.
- El avatar ahora valida maximo 5 MB igual que la interfaz.
- La hidratacion del usuario local ahora contempla banner.
- ArtistPage vuelve a cargar reseñas cuando cambia el token de sesion.
- ProPage ahora muestra feedback visible si falla la preferencia de pago.
- Admin envia fechas como ISO explicito para evitar corrimientos por timezone.
- SearchPage evita duplicar la misma busqueda al enviar el formulario.

Impacto:
- Menos estados parciales.
- Mejor continuidad del funnel PRO.
- Menos desincronizacion visual en perfil y reseñas.
- Menos requests redundantes.
