CAMBIOS DE SEGURIDAD Y CORRECCIONES
Fecha: 28/03/2026

Documento reorganizado por tipo de arreglo para dejar mas claro que cambios quedaron del lado del frontend, backend, base de datos y configuracion general.


1. FRONTEND

1.1. Autenticacion y verificacion

Problemas detectados:
- El login, el registro y el reenvio del mail de verificacion dependian de flujos que no siempre mostraban errores claros al usuario.
- El frontend habia quedado atado a un flujo de verificacion que esperaba tokens propios del backend.
- Si el registro fallaba por configuracion o por reenvio, la interfaz no siempre podia distinguir bien la causa.

Cambio aplicado:
- El frontend usa los endpoints del backend para login, registro y reenvio de verificacion.
- Se agregaron mensajes de error mas claros para fallos de Supabase Auth y para lookup de reenvio.
- La pagina /verify-email ahora acepta dos escenarios:
  - el flujo viejo con ?token=
  - el redirect nuevo de Supabase Auth sin depender del token propio del backend

Impacto:
- La interfaz deja de quedar muda cuando falla el envio o el reenvio.
- El enlace que llega desde Supabase ahora puede completarse correctamente en la pagina de verificacion.
- Se reduce la confusion del usuario final cuando la cuenta existe pero aun no esta confirmada.


1.2. Perfil, banner y consistencia visual

Problemas detectados:
- Varias ediciones de perfil podian quedar a mitad de camino si fallaba el segundo paso.
- El banner podia no reaparecer correctamente despues de login o recarga.
- El modal de perfil podia conservar previews viejas al cerrarse.

Cambio aplicado:
- Las actualizaciones de username, telefono y avatar se consolidaron desde backend.
- Si un upload de avatar o banner falla al final, se intenta limpiar el archivo recien subido.
- La hidratacion del usuario local contempla banner.
- El cierre del modal limpia avatar y banner pendientes en los caminos relevantes.

Impacto:
- Menos estados parciales.
- Menos basura en storage por uploads fallidos.
- Menos desincronizacion visual en perfil.


1.3. Navegacion, busqueda y experiencia de uso

Problemas detectados:
- El flujo de compra PRO podia llevar al login y luego mandar siempre a /profile.
- La busqueda manual podia duplicar requests.
- SearchPage podia mostrar resultados viejos si dos respuestas llegaban fuera de orden.
- El autocomplete del navbar podia mostrar sugerencias stale al escribir rapido.
- ArtistPage podia dejar resenas y likes desactualizados al cambiar la sesion.
- ProPage casi no mostraba feedback si fallaba la creacion del pago.
- Los filtros de fecha del admin podian correrse por timezone.

Cambio aplicado:
- El login respeta redirectTo y vuelve a /pro cuando corresponde.
- SearchPage evita requests duplicados y descarta respuestas atrasadas.
- El autocomplete del navbar descarta respuestas viejas y limpia mejor su estado.
- ArtistPage vuelve a cargar resenas cuando cambia el token de sesion.
- ProPage muestra feedback visible si falla la preferencia de pago.
- Admin envia fechas como ISO explicito.

Impacto:
- Mejor continuidad del funnel PRO.
- Menos requests redundantes.
- Menos resultados incoherentes al buscar rapido.
- Menos desincronizacion visual en paginas de contenido.


2. BACKEND

2.1. Rate limiting real sobre el flujo de autenticacion

Problema detectado:
- Parte del flujo de autenticacion no siempre quedaba cubierto por los limites del backend.

Cambio aplicado:
- Login, registro y reenvio de verificacion quedaron cubiertos por endpoints con rate limiting efectivo.

Mensajes expuestos al cliente:
- AUTH_LOGIN_RATE_LIMITED
- AUTH_REGISTER_RATE_LIMITED
- AUTH_VERIFY_EMAIL_RESEND_RATE_LIMITED

Impacto:
- Reduce brute force.
- Reduce abuso del registro.
- Reduce spam de reenvios de verificacion.


2.2. Verificacion de email delegada a Supabase Auth

Problema detectado:
- El backend mantenia un flujo propio de verificacion y el envio podia no ocurrir aunque el registro devolviera exito.
- El proyecto no queria enviar ese mail con Resend ni con otro proveedor propio del backend, sino desde Supabase.
- El backend podia no ver la configuracion publica de Supabase si esa variable existia solo en el .env raiz.

Cambio aplicado:
- El registro local ahora crea la cuenta en Supabase Auth mediante signUp.
- El reenvio de verificacion ahora usa supabase.auth.resend.
- El backend toma como fallback el .env raiz para leer configuracion compartida si hace falta.
- Se agrega uso de SUPABASE_ANON_KEY o VITE_SUPABASE_ANON_KEY para poder hablar con Supabase Auth desde el backend.
- Si falta esa configuracion, ahora se devuelve un error explicito en vez de fallar en silencio.

Impacto:
- El mail de verificacion pasa a depender del sistema correcto: Supabase Auth.
- Si el correo no sale, el siguiente punto a revisar ya no es el codigo del envio local sino la configuracion de Supabase.
- El usuario recibe feedback real si falta configuracion o si el reenvio falla.

Estado esperado ahora:
- Usuario se registra.
- El backend crea la cuenta en auth.users.
- Supabase envia el mail de confirmacion.
- La tabla users de la app se sincroniza con ese usuario.
- Si el usuario aun no confirmo, el login sigue bloqueado.
- Una vez confirmado, el login puede avanzar y el backend actualiza el estado local si hacia falta.


2.3. Compatibilidad de login para cuentas heredadas

Problema detectado:
- Existian cuentas con auth_provider = local pero sin password_hash en users.
- Eso hacia que /api/auth/login devolviera 401 incluso para usuarios validos.

Cambio aplicado:
- El login intenta primero el hash local.
- Si el hash falta o no coincide, prueba un fallback controlado contra Supabase Auth.
- Si el usuario entra por ese fallback, se reescribe un password_hash valido en users.
- Ademas, si el usuario ya verifico su mail en Supabase pero la tabla local seguia vieja, el login re-sincroniza ese estado antes de bloquearlo.

Impacto:
- Usuarios heredados pueden volver a iniciar sesion.
- Menos cuentas rotas por desincronizacion entre users y auth.users.


2.4. Restriccion PRO para banner

Problema detectado:
- La restriccion "solo PRO puede subir banner" dependia demasiado del frontend.

Cambio aplicado:
- La actualizacion de banner pasa por backend.
- El backend rechaza el cambio si el usuario no es PRO.

Impacto:
- La restriccion comercial queda reforzada del lado servidor.


2.5. Borrado real de usuarios desde admin

Problemas detectados:
- El panel admin podia borrar solo la fila local y dejar viva la cuenta en Auth.
- En algunos casos el borrado fallaba por registros relacionados no limpiados.
- Algunas cuentas heredadas no existian ya en Supabase Auth y ese 404 se trataba como error fatal.

Cambio aplicado:
- El borrado admin limpia registros relacionados del usuario antes de eliminar la cuenta de Auth.
- Tambien elimina email_verification_tokens del usuario.
- Se tolera user_not_found y status 404 como casos validos cuando la cuenta ya no existe en Auth.

Impacto:
- El borrado admin es mas coherente.
- Menos fallos por relaciones colgantes.
- Menos errores falsos en cuentas huerfanas o heredadas.


2.6. Hardening del correo HTML y de los tokens

Problemas detectados:
- Valores controlados por usuario podian interpolarse en el HTML del correo.
- Los tokens de verificacion se guardaban en texto plano.

Cambio aplicado:
- Se escaparon valores antes de insertarlos en la plantilla HTML.
- Los nuevos tokens de verificacion se guardan hasheados con SHA-256.
- La verificacion sigue aceptando tokens viejos en texto plano para no romper enlaces emitidos antes del cambio.

Impacto:
- Menor riesgo de inyeccion de markup en correos.
- Menor riesgo si alguien accede a la tabla de tokens.

Nota:
- Aunque el flujo principal nuevo de verificacion pasa por Supabase Auth, este hardening se mantiene para compatibilidad con enlaces antiguos y para no romper datos ya existentes.


2.7. Dependencias y mantenimiento

Problema detectado:
- npm audit habia marcado una vulnerabilidad en path-to-regexp usada por el backend.

Cambio aplicado:
- Se fijo una version corregida mediante override.
- Se actualizo el package-lock del backend.

Impacto:
- Se elimina el hallazgo reportado por npm audit para esa cadena de dependencias.


3. BASE DE DATOS

3.1. Tablas y columnas usadas por autenticacion

Estado cubierto por los cambios:
- users mantiene estado local de verificacion:
  - is_verified
  - verified_at
  - verification_sent_at
- email_verification_tokens sigue existiendo para compatibilidad con el flujo anterior.
- profiles sigue sincronizado con usuarios autenticados.

Impacto:
- La app conserva compatibilidad con datos existentes mientras migra el flujo principal hacia Supabase Auth.


3.2. Rate limiting persistente

Cambio aplicado:
- Se dejaron scripts de Supabase para sostener el rate limiting del backend:
  - supabase/create_rate_limits.sql
  - supabase/fix_consume_rate_limit_signature.sql

Impacto:
- El modo deseado en produccion es el respaldado por Supabase, no solo memoria local.


4. CONFIGURACION Y OPERACION

4.1. Cosas a verificar en Supabase para el mail de verificacion

Para que el flujo nuevo funcione de punta a punta, debe cumplirse esto:
- Email Confirmations habilitado en Supabase Auth.
- Redirect URL permitida para /verify-email.
- Proveedor de correo configurado dentro de Supabase si no se usa el sender por defecto del entorno.
- Revisar spam si el mail sigue sin verse en bandeja principal.


4.2. Variables y secretos

Consideraciones:
- .env no debe subirse al repositorio.
- El backend ahora necesita poder leer SUPABASE_ANON_KEY o VITE_SUPABASE_ANON_KEY para delegar signUp y resend a Supabase Auth.
- El webhook de Mercado Pago debe seguir teniendo configurado su secreto en produccion.


4.3. Estado general operativo

Estado final esperado despues de deploy:
- Login, registro y resend pasan por backend.
- El rate limiting protege el flujo real.
- El mail de verificacion sale desde Supabase Auth.
- El borrado admin deja de ser superficial.
- El banner PRO queda reforzado del lado servidor.
- Las cuentas heredadas pueden repararse al iniciar sesion.

Advertencia tecnica no bloqueante:
- El build sigue mostrando warning de Vite por chunk principal grande. No rompe funcionalidad, pero conviene hacer code splitting mas adelante.

Estado confirmado en esta pasada:
- No se detectaron errores funcionales criticos abiertos.
- No se detecto una rotura general del frontend ni del backend.
- Login, registro, resend, rankings, search, artist, album, reviews y ratings respondieron bien en las pruebas hechas.

Resumen ejecutivo:
